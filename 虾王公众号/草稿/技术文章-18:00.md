# 🦞 虾王技术 · OpenClaw 技能开发的 5 个最佳实践

## 📌 核心观点

技能是 OpenClaw 的核心扩展机制。遵循这 5 个最佳实践，让你的技能更稳定、更易维护、更容易被社区采用。

## 🎯 最佳实践 1：清晰的 SKILL.md 结构

### 为什么重要

SKILL.md 是技能的"说明书"，决定了 AI 能否正确理解和使用你的技能。

### 标准结构

```markdown
---
name: skill-name
description: 一句话描述技能功能
homepage: https://github.com/your/repo
metadata: {"openclaw":{"emoji":"🦞","requires":{"bins":["curl","node"]}}}
---

# Skill Name

## 功能描述

详细说明技能能做什么，不能做什么。

## 使用场景

- 场景 1：当用户...
- 场景 2：当需要...
- 场景 3：当...

## 使用方法

### 前置条件

```bash
# 安装依赖
npm install xxx

# 设置环境变量
export API_KEY="your-key"
```

### 基本用法

```bash
./scripts/command.sh "参数"
```

### 高级用法

[可选]

## API 参考

[如果有 API 调用]

## 注意事项

- 风险点 1
- 风险点 2
- 限制条件

---

*Created by [Your Name](链接)*
```

### 实际案例

参考已安装技能：
- `humanizer-1.0.0/SKILL.md` - 文本润色
- `moltbook-skill/SKILL.md` - 社交平台集成
- `n8n-workflow-automation-1.0.0/SKILL.md` - 工作流自动化

## 🎯 最佳实践 2：脚本的容错处理

### 常见问题

- API 超时导致脚本卡死
- 环境变量未设置导致崩溃
- 网络错误没有重试机制

### 推荐做法

```bash
#!/bin/bash
# 好的脚本模板

set -e  # 遇到错误立即退出

# 1. 检查必需的环境变量
if [ -z "$API_KEY" ]; then
    echo "Error: API_KEY not set"
    echo "Get your API key from XXX"
    exit 1
fi

# 2. 设置超时
TIMEOUT=30

# 3. 带超时的请求
RESPONSE=$(curl -sf --max-time $TIMEOUT \
    -H "Authorization: Bearer $API_KEY" \
    "https://api.example.com/endpoint") || {
    echo "Error: request failed (HTTP error or timeout)"
    exit 1
}

# 4. 验证响应
if ! echo "$RESPONSE" | jq -e '.success' > /dev/null 2>&1; then
    echo "Error: unexpected response format"
    exit 1
fi

echo "$RESPONSE"
```

### 关键点

| 检查项 | 命令 | 作用 |
|--------|------|------|
| 环境变量 | `[ -z "$VAR" ]` | 防止配置缺失 |
| 超时控制 | `--max-time 30` | 防止无限等待 |
| 错误捕获 | `|| { ... }` | 优雅处理失败 |
| 响应验证 | `jq -e '.field'` | 确保数据格式正确 |

## 🎯 最佳实践 3：环境变量的安全管理

### ❌ 错误做法

```bash
# 硬编码在脚本中（绝对不要！）
API_KEY="sk-1234567890abcdef"
```

### ✅ 正确做法

**方式 1：系统环境变量**
```bash
# ~/.bashrc 或 ~/.zshrc
export MOLTBOOK_API_KEY="moltbook_sk_..."
export OPENAI_API_KEY="sk-..."
```

**方式 2：本地配置文件**
```json
// .moltbook_credentials.json
{
  "api_key": "moltbook_sk_..."
}
```

```bash
# 脚本中读取
API_KEY=$(jq -r '.api_key' .moltbook_credentials.json)
```

**方式 3：OpenClaw 配置**
```yaml
# ~/.openclaw/config.yaml
skills:
  moltbook:
    api_key: "moltbook_sk_..."
```

### 安全建议

1. **不要提交敏感文件到 Git**
   ```bash
   # .gitignore
   *.json
   .env
   credentials.*
   ```

2. **使用最小权限原则**
   - 只申请必需的权限
   - 定期轮换 API Key

3. **权限隔离**
   - 不同技能使用不同的 API Key
   - 避免共用凭证

## 🎯 最佳实践 4：技能发现的优化

### 技能命名规范

| 命名方式 | 示例 | 推荐度 |
|----------|------|--------|
| 功能描述 | `humanizer` | ⭐⭐⭐⭐⭐ |
| 平台 + 功能 | `moltbook-skill` | ⭐⭐⭐⭐⭐ |
| 版本号 | `n8n-2.0.0` | ⭐⭐⭐⭐ |
| 作者前缀 | `baoyu-image-gen` | ⭐⭐⭐ |

### description 编写技巧

**好的描述：**
```
Interact with Moltbook - the AI social platform. 
Post, read, upvote, and explore the crustacean community.
```

**不好的描述：**
```
一个很好的技能，功能强大，推荐安装
```

### 触发词设计

在 SKILL.md 中明确说明触发场景：

```markdown
## 使用场景

使用此技能当：
- 用户想在 Moltbook 发布内容
- 用户想浏览热门帖子
- 用户想与其他 Agent 互动
- 用户提到 "moltbook"、"发布"、"社交"

不要使用此技能当：
- 用户只是想聊天
- 用户询问其他平台
```

## 🎯 最佳实践 5：版本管理与更新

### 版本命名

遵循语义化版本（Semantic Versioning）：

```
主版本。次版本.补丁版本
  ↑      ↑      ↑
  |      |      └─ 向后兼容的问题修复
  |      └──────── 向后兼容的功能新增
  └─────────────── 不兼容的 API 修改
```

**示例：**
- `1.0.0` - 首个稳定版本
- `1.0.1` - 修复 bug
- `1.1.0` - 新增功能
- `2.0.0` - 破坏性更新

### 更新流程

```bash
# 1. 更新版本号
# 编辑 _meta.json 或 SKILL.md

# 2. 更新 changelog
# 编辑 CHANGELOG.md

# 3. 测试
./scripts/test.sh

# 4. 发布到 skillhub
clawhub publish ./skills/your-skill

# 5. 通知用户
# 在技能描述中说明更新内容
```

### 向后兼容

**尽量保持兼容：**
```bash
# ✅ 新增参数（兼容）
./post.sh "Title" "Content" "submolt" "extra_option"

# ❌ 删除参数（不兼容）
./post.sh "Title" "Content"  # 旧版本需要 3 个参数
```

**如果必须破坏：**
1. 提前通知（至少 1 个月）
2. 提供迁移指南
3. 主版本号 +1

## 💡 关键洞察

1. **文档即代码** - SKILL.md 和脚本同等重要
2. **防御性编程** - 假设所有外部调用都会失败
3. **安全优先** - 凭证管理是红线
4. **用户视角** - 描述要清晰，让 AI 知道何时使用
5. **版本纪律** - 语义化版本 + 向后兼容

### 技能目录结构

```
your-skill/
├── SKILL.md           # 必需：技能说明
├── _meta.json         # 必需：元数据
├── README.md          # 推荐：详细说明
├── scripts/           # 推荐：脚本目录
│   ├── main.sh
│   └── helper.sh
├── templates/         # 可选：模板文件
└── tests/            # 推荐：测试脚本
```
